# 爬虫运行记录分析

## 职责

分析 `main.py` 爬虫的终端日志输出，生成标准化的运行记录条目，写入 `doc/runlog.md`。

## 触发场景

- 用户粘贴爬虫日志（含进度条、分类分布表）
- 用户要求"记录运行状态""更新 runlog""分析日志"
- 爬虫出现错误/崩溃，需要记录错误分析和改进方案
- 用户提供 checkpoint 数据或进度截图

## 执行步骤

### 步骤 1：解析输入

从日志中提取以下关键字段：
- **时间戳**：日志行首的 `YYYY-MM-DD HH:MM:SS`
- **累计数量**：`进度: N/100000` 中的 N
- **分类分布**：`A: ... B: ... C: ...` 各行中的 count/target
- **当前标签**：`[douban] 标签名 → CLC码 (置信度)`
- **速率**：从 Recent Speed 或手动计算
- **错误信息**：PermissionError / 418 / timeout 等

### 步骤 2：计算统计量

- **完成率**：每个分类 `已采集 / 目标 × 100%`
- **增量变化**：与上一个 runlog 条目比较各分类的 delta
- **速率**：`新增量 / 时间间隔`（条/小时）
- **ETA**：`剩余量 / 速率`，换算为日期，保守取整

### 步骤 3：识别关键事件

- 哪个分类出现了**突破性增长**（从 0→显著数量）
- 哪些分类**停滞**（delta=0 或接近 0）
- 速率相比上一时段是**上升还是下降**
- 是否有**错误/崩溃**事件

### 步骤 4：写入 runlog.md

严格按照 `assets/entry-template.md` 格式写入，包含所有必填字段。

## 输出标准

参见 `references/format-standards.md` 中的完整规范。核心要求：

1. 标题格式：`## YYYY-MM-DD HH:MM — N条[（状态描述）]`
2. 必含字段：运行时长、数据源、累计采集、速率、剩余、预计完成
3. 分类表格按目标完成率降序排列（已完成类在上，零进度类在下）
4. 分析段至少包含 3 条洞察
5. 状态明确标注："运行中" 或 "已停止"

## 错误处理

当遇到错误/崩溃时，额外记录：
- **错误类型**和完整错误消息
- **原因分析**：根本原因
- **影响评估**：丢失多少数据、回退多少进度
- **恢复步骤**：具体可执行的命令
- **改进方案**：防止再次发生的代码修改建议，记录到对应的 doc 文件中

## 注意事项

- 所有输出为中文
- 分类表使用 22 个完整大类（A-Z），即使某类为 0 也必须列出
- ETA 预测使用保守估计（当前速率不再下降为前提）
- 不要修改正在运行的代码，只记录和分析
- 写入前先读取现有 runlog.md 确认上一个条目的数据用于计算 delta
