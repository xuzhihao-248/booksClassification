# 中文图书数据采集系统 — 需求文档

## 1. 项目概述

### 1.1 项目目标

从互联网采集 **10万条** 中文图书数据，用于训练中文文本分类器。每条数据包含三栏核心信息：**书名、内容简介、中图法分类号**。数据需覆盖多个学科类别以保证分类器的泛化能力。

### 1.2 输出规格

| 项目 | 规格 |
|------|------|
| 数据量 | 100,000 条 |
| 核心字段 | 书名(title)、内容简介(description)、中图法分类号(clc_code) |
| 辅助字段 | 作者、出版社、出版年、ISBN、豆瓣标签、豆瓣评分、来源URL |
| 文件格式 | Excel (.xlsx)，UTF-8编码 |
| 分类标准 | 中图法 (CLC, Chinese Library Classification) 22大类 |

### 1.3 分类覆盖

目标覆盖中图法全部 22 个一级大类，各类别目标数量根据图书市场实际分布设定：

| B | 哲学、宗教 | 5,000 | O | 数理科学和化学 | 3,000 |

## 2. 技术方案

### 2.1 数据源选择

**主数据源：豆瓣读书 (book.douban.com)**

选择理由：
- 中文图书数据最丰富，涵盖各学科门类
- 拥有完善的用户标签体系，可作为分类依据
- 每本书有独立详情页，包含书名、简介、标签等完整信息
- 开放匿名访问（需合理控制频率）

### 2.2 分类方案

豆瓣不直接提供中图法分类号，采用**标签映射法**：

```
豆瓣用户标签 → tag_to_clc.json 映射表 → 中图法分类号
```

映射策略（三级）：
1. **精确匹配**：标签在映射表中直接命中，置信度最高
2. **模糊匹配**：通过 `SequenceMatcher` 计算字符串相似度（阈值 0.80），降权采纳
3. **多标签投票**：每本书通常有 5-10 个标签，各自投票，加权计算最终分类
4. **兜底分类**：无法匹配时归入 Z 类（综合性图书）

### 2.3 技术栈

| 组件 | 技术选型 | 用途 |
|------|----------|------|
| HTTP 客户端 | `httpx` | 支持 HTTP/2、连接池、超时控制 |
| HTML 解析 | `BeautifulSoup4` + `lxml` | CSS选择器提取结构化数据 |
| UA 轮换 | `fake-useragent` | 自动生成真实浏览器UA |
| Excel 输出 | `openpyxl` | 生成 .xlsx 文件 |
| 去重存储 | `sqlite3`（内置）| 持久化已爬取ID |
| 日志 | `logging`（内置）| 运行日志记录 |

### 2.4 系统架构

```
main.py  (单文件，内含以下模块)
├── AntiScrapeSession    — HTTP会话管理 + 反爬措施
├── BookParser           — HTML页面解析
├── CLCMapper            — 标签→中图法分类映射
├── CheckpointManager    — 断点续传状态管理
├── ExcelWriter          — 缓冲写入Excel
└── BookScraper          — 主调度器，协调全部流程
```

## 3. 反爬策略

| 层级 | 措施 | 实现方式 |
|------|------|----------|
| L1 | 请求间隔 | 每次请求后随机等待 2-5 秒，含 30% 抖动 |
| L2 | UA 轮换 | 50+ User-Agent 池，每次请求随机选取 |
| L3 | 请求头伪装 | Accept、Accept-Language、Sec-Fetch 等与 UA 匹配 |
| L4 | Referer 链 | 从标签列表页进入详情页时携带来源URL |
| L5 | 指数退避 | 429/403/5xx 重试最多5次，间隔指数递增 |

## 4. 容错与恢复

- **断点续传**：每1000条自动保存进度（已爬取数、分类计数、最后ID）
- **去重持久化**：SQLite 存储已爬取的豆瓣ID，重启不丢失
- **优雅中断**：Ctrl+C 触发保存，数据不丢失
- **解析容错**：单个页面解析失败不影响整体流程，记录日志后跳过
- **空值兜底**：简介缺失时填充空字符串，标签缺失时归入Z类

## 5. 运行计划

| 阶段 | 规模 | 目的 |
|------|------|------|
| 测试运行 | 100 条 | 验证解析和分类全流程 |
| 中等运行 | 5,000 条 | 验证分类分布和反爬稳定性 |
| 正式运行 | 100,000 条 | 完整数据采集 |

预计正式运行时间：约 4 天（100k × 3.5s/条 ÷ 86400s/天）

## 6. 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| IP被豆瓣封禁 | 中 | 高 | 保守延迟、低并发、预留代理接口 |
| 豆瓣页面结构变更 | 中 | 中 | 多选择器回退、监控解析成功率 |
| 冷门类别数据不足 | 高 | 中 | 定向关键词搜索补充、适当降低目标 |
| 标签映射精度不足 | 中 | 中 | 人工抽查校准、置信度标记 |
| 网络不稳定 | 低 | 低 | 断点续传、重试机制 |
